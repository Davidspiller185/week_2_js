1. כשכל אחד משנה שורה אחרת באותו קובץ ואז git יכול  לשלב בינהם
2. אם שניהם עשו שינוי את אותה שורה לערכים שונים אז יהיה merge conflict
3. git מחליט בעצמו כשכל אחד עשה שינוי בשורה שונה ואז המיזוג קורה אוטומטי אבל אם כל אחד עשה שינוי באותה שורה אז הוא לא יודע למזג ומה למזג ולכן יש merge conflict
4. merge conflict זה לא טעות זה בסך הכל מצב שבו git לא יודע איך לשלב את השינויים במקביל זה מצב די מקובל כשעובדים בשיתוף.
5. השורה הספציפית שהשתנתה היא זאת שקובעת את הmerge conflict
6. HEAD >>>>>>> מסמן את תחילת השינויים
======= מפריד בין שתי הגירסאות של הקוד
feature/api-url <<<<<<< מסמן את סוף השינויים של הbranch שמנסים למזג.
7. <<<<<<< HEAD
TIMEOUT = 30
=======
TIMEOUT = 60
>>>>>>> feature/api-url
TIMEOUT = 30 הגיע מהbranch הנוכחי HEAD
TIMEOUT = 60 הגיע מהfeature/api-url שמנסים למזג
8. resolve conflict להחליט איזו גרסה של הקוד תישאר בפועל צריך לבחור אחד מהגירסאות ואז למחוק את כל הסימנים של conflict ולשמור את הקובץ ולבצע git add git commit
9. אם היו משאירים את הConflict markers  אז הcommit היה שומר את הקובץ עם הסימונים עצמםואז זה יגרום לשגיאות או שיאר טקסט לא קשור לכן חייבים להסיר את הconflictMarkers
10. pull request בקשה למיזוג שינויים בין שני branch דרך gitub pull request מאפשר לבדוק את הקוד ולאשר ולבדוק אם המיזוג אפשרי לעומת זאת git merge ממזג את הbranch ישירות בלי תהליך של בדיקה.
11. base branch - הbranch שאליו רוצים למזג את השינויים
compare branch - branch שממנו מגיעים השינויים
12. code review תהליך שבו מפתח אחר בודק את הקוד לפני המיזוג.
13. לפי התהליך יש לעשות code review ואז לאשר את הpoolrequest שיאשר לי את המיזוג ואז לבצע את המיזוג
14. pull request יש לו יתרון מאשר git merge ישיר כיוון שהוא מוסיף שלב של בדיקה ואישור לפני המיזוג 
15. פתרון conflict: פותחים את קובץ הconflict מוצאים את הconflict Markers ומחליטים איזה גירסה להשאיר ומוחקים את כל הconflict Markers ושומרים את הקובץ ומריצים git add ואז git יודע שהסתירה נפתרה
16. עדיין צריך להשלים עם git commit 
17. פתרון שגוי של Conflict זה יגרום לאיבוד שינויים כי אם נבחר בגירסה הלא נכונה ואז השינוי של הקובץ הנכון לא יכלל בגירסה הסופית.
18. אחררי המיזוג את נראה בהיסטוריית הcommits גם את השינויים שהגיעו מfeature/checkout-flow
19. לפתוח poolrequest בgitub וללחוץ על New pool request ולבחור את הbase branch ואת הcompare branch עם תיאור הפיצ'ר ואז עושים code review כדי למצוא אפ יש באגים לגילוי conflict פותחים את הקובץ מגלים את הconflict marker בוחרים או משלבים בין גירסאות  מוחקים את הסימונים ושומרים את הקובץ ומבצעים git add git commit
אחרי שהpoolrequest קיבל approve אז לוחצים על merge pull request ואז השינויים נכנסים ומתמזגים ואז אחרי מיזוג מוצלח מקובל למחוק את ה branch - git branch -d feature/user-avatar
20. merge conflict   יכול לקרות גם בלי pr זה נעשה בין אם הmerge נעשה בטרמינל ובין אם זה Pr בטרמינל
21. החיסרון של pr שהוא מוסיף שלבים לעבודה שזה cr אישור ומיזוגלכן לדברים קטנים לא עושים pr
22. merge conflict - מונע מgit למזג אוטומטית איפה שיש סתירות pr- מוסיף תהליך של Codereview ואישור
